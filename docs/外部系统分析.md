## 1. 各服务职责与目录结构概览

### 1.1 business 服务（端口 7080）

**职责**：对外 API 网关层，提供前端（碳管理/能效/设备管理）所需的接口。封装底层业务逻辑，调用 ems-cloud-service 和 hvac-iot。

**目录结构**：
```
com.csot.business/
├── controller/    # 对外 REST 接口
├── service/       # 业务逻辑
├── entity/        # JPA 实体（与 MySQL 表映射）
├── repository/    # JPA Repository（数据访问）
├── mapper/        # MyBatis Mapper（极少使用）
├── dto/           # 传输对象
├── config/        # 配置类
├── util/          # 工具类
├── api/           # Feign 客户端（需确认）
```

### 1.2 ems-cloud-service 服务（端口 8030）

**职责**：核心数据服务层。管理设备/模型/参数的元数据（MySQL），以及时序数据（ClickHouse）。提供 ClickHouse 查询接口、参数管理 API。

**目录结构**：
```
com.ems/
├── controller/    # REST 接口
│   ├── parameter/ # 参数管理相关
│   └── schedule/  # 定时任务
├── service/       # 业务逻辑（IClickHouseUsedService 等）
├── entity/        # JPA 实体
├── mapper/        # MyBatis Mapper
│   ├── mapper/    # MySQL Mapper
│   ├── ckmapper/  # ClickHouse Mapper
├── dto/           # 传输对象
├── config/        # 配置类
├── mqhandler/     # Kafka/MQTT 消息处理
```

### 1.3 hvac-iot / service-model 服务（端口 8048）

**职责**：暖通 IoT 设备管理。管理设备注册、指令下发、测点绑定、数据模拟/采样。提供设备实例管理、Cmd下发接口。

**目录结构**：

## 3. 测点参数的完整生命周期链路

### 3.1 参数定义流程

```
1. 定义物模型（设备模板）
   → device_model 表
   → model_mark 是核心标识（如 "eleMeter_virtual"）
   
2. 定义物模型参数
   → param_info 表
   → 每个模型下可以有 N 个参数
   → 关键字段：model_mark + param_mark → 全局唯一定位一个"参数槽位"
   
3. 创建设备实例
   → device_info 表
   → 属于某个 model（device_info.model_mark）
   → device_mark 实例级别标识（如 "eleMeter_virtual_001"）
   
4. 绑定物理测点（可选）
   → device_param_point_mapping 表
   → 将逻辑参数（param_mark）绑定到物理测点（device_model_point_info）
   → 支持公式转换（虚点可通过公式计算）
   → is_virtual=1 表示虚点（非直接采集）
```

### 3.2 前端选择参数 → 读取实时值的完整链路

#### 步骤 1：获取模型列表

```
调用 business 服务：
POST /api/business/device/queryModelList
请求体：{"tenantId": "geek", "category": ""}
返回：设备模型列表（包含 modelMark、modelName、category 等）
→ 从 device_model 表查询
```

#### 步骤 2：获取某模型下的设备列表

```
调用 business 服务：
GET /api/business/device/queryDeviceListByModelMark?modelMark=xxxx
→ 从 device_info 表查询
```

#### 步骤 3：获取某模型的参数列表

```
调用 business 服务：
GET /api/business/device/queryParamlist?modelMark=xxxx&paramType=xxxx&search=xxxx
→ 从 param_info 表查询（filter by model_mark）
```

#### 步骤 4：获取参数实时值

**方式 A：从 Redis 读取最新值（最快）**
```
POST /api/business/device/realdata
请求体：{"params": "eleMeter_virtual#DM-001#energy_used,..."}
返回：
[
  {"deviceMark":"DM-001","paramMark":"energy_used","value":"123.45","updateTime":"2026-05-12 17:30:00"},
  ...
]
后端逻辑：
→ 拼接 Redis key：{tenantId}_REDIS_REPLACE_FLAG_{modelMark}_REDIS_REPLACE_FLAG_{deviceMark}_REDIS_REPLACE_FLAG_{paramMark}
→ 从 DP_LAST_DATA (Hash) 中获取
```

**方式 B：从 ClickHouse 查询历史**

```
调用 ems-cloud-service：
POST /api/cms-cloud-service/clickHouse/queryByParam
请求体：{
  "tenantId": "geek",
  "modelMark": "eleMeter_virtual",
  "deviceMark": "eleMeter_virtual_001",
  "paramMark": "energy_used",
  "startDate": "2026-05-12 00:00:00",
  "endDate": "2026-05-13 00:00:00",
  "resultDateType": "9"  // 9=实时(每分钟)
}
→ 从 {tenantId}_device_param_all 表查询
→ 支持多种粒度：年/月/日/时/30分钟/15分钟/5分钟/实时
```

**方式 C：批量查询最新值**

```
调用 ems-cloud-service：
POST /api/cms-cloud-service/comm-raw-param/batch-latest
请求体：[
  {"tenantId":"geek","deviceMark":"xxx","modelMark":"xxx","paramMark":"xxx"},
  ...
]
→ 从 comm_raw_param_last 表查询
```

#### 步骤 5：下发指令

```
调用 hvac-iot 服务：
POST /api/hvac_iot/cmd
请求体：[
  {
    "deviceMark": "Chiller-01",
    "paramMark": "ChAmpLmtSetting",
    "value": "50",
    ...
  }
]
→ 已确认在 CmdController 中实现
```

### 3.3 物模型（modelMark）和实例（deviceMark）的关系

```
device_model（物模型/模板）
    ↓ modelMark（一个模型可以有多个设备实例）
device_info（设备实例）
    ↓ deviceMark（每个实例有唯一标识）

例如：
  modelMark = "eleMeter_virtual"    ← 虚拟电表这个模板
  deviceMark = "eleMeter_virtual_001"  ← 第1个虚拟电表实例
  deviceMark = "eleMeter_virtual_002"  ← 第2个虚拟电表实例
```

### 3.4 参数标识（paramMark）的定义规则

**paramMark** 是在一个物模型（modelMark）内部唯一的参数标识字符串。

- 由业务定义，无固定前缀规则
- 通常使用有意义的英文名或缩写
- 示例：`energy_used`、`Ia`（A相电流）、`max_demand`、`ChAmpLmtSetting`
- **全局唯一性**：`modelMark + deviceMark + paramMark` 三元组唯一确定一个测点的值

### 3.5 缓存机制

- **Redis**：最新一次上报的值缓存在 `DP_LAST_DATA` Hash 中
- **ClickHouse comm_raw_param_last**：ClickHouse 层面的快照表，记录每个参数的最新值
- **Redis vs ClickHouse**：Redis 用于毫秒级读取最新值；ClickHouse 用于历史时序分析

## 5. cms-rule 复用建议

### 5.1 方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| **A. 调接口读数据** | 解耦，不依赖数据库权限 | 性能瓶颈，依赖网络 | 低频查询，少量参数 |
| **B. 直连数据库** | 高性能，灵活 | 需数据库权限，耦合度高 | 高频规则计算 |
| **C. 数据复制（同步到 cms-rule 自己的库）** | 隔离性好 | 维护同步逻辑，数据延迟 | 需要稳定规则数据 |

### 5.2 建议方案

#### 对元数据（模型、设备、参数定义）

**推荐方案 B（直读 MySQL）**，原因：
- 数据量小（参数定义变化不频繁）
- 无需订阅数据流
- cms-rule 本就在同样的 Java/Spring Boot 技术栈

需要获取的表：
- `device_model` — 获取所有物模型
- `device_info` — 获取设备列表
- `param_info` — 获取参数定义
- `device_param_point_mapping` — 获取参数↔测点绑定关系

#### 对实时/时序数据

**推荐方案 A + C 混合**：
1. **实时值（秒级）**：使用 Redis 读取最新值（`DP_LAST_DATA` 或 `comm_raw_param_last` API）
2. **历史值（分钟级~）**：使用 ClickHouse API `/api/cms-cloud-service/clickHouse/queryByParam` 或 `/api/cms-cloud-service/clickHouse/queryByParamAccumulated`
3. **高频规则计算**：如果需要频繁访问同一批测点数据，建议复制到 cms-rule 库（或使用 ClickHouse 本地表）

#### 对 Redis 的考虑

Redis 中 `DP_LAST_DATA` 是业务服务的缓存。cms-rule 可以考虑：
- 直接读同一个 Redis
- 或者调用 `/api/business/device/realdata` 接口
- 或者使用 `/api/cms-cloud-service/comm-raw-param/batch-latest` API（更底层，不依赖 Redis）

后者是最推荐的方式，因为：
- 不依赖特定 Redis Key 格式
- 通过 comm_raw_param_last ClickHouse 表读取，数据可靠
- 支持批量查询，性能好

### 5.3 数据模型映射

```
cms-rule 需要：
  - 物模型标识: modelMark
  - 设备标识: deviceMark  
  - 参数标识: paramMark
  → 三元组定位一个测点值

外部系统数据路径：
  modelMark + deviceMark + paramMark
      ↓
  lookup device_param_point_mapping
      ↓
  获取物理测点 (device_model_point_info_id)
      ↓  或 直接
  从 comm_raw_param_last / device_param_all 获取值
```

### 5.4 注意事项

1. **参数值来源**：`valueSource` 字段区分"表计值（sign）"和"增加值（add）"，规则计算时需根据值类型做不同处理
2. **虚点（virtual）**：device_param_point_mapping.is_virtual=1 的参数可能通过公式计算得到，需要先解析公式
3. **下发标记**：只有 deliverFlag=2 的参数才能下发，cms-rule 如果需要下发指令需检查此标记
4. **租户隔离**：所有表都按 `tenant_id` 隔离，默认为 "geek"
5. **数据源配置**：dev 环境的 MySQL/ClickHouse/Redis 连接配置见各服务的 application-dev.yml，实际的端口号和密码请确认
6. **ClickHouse 分表**：时序数据按租户分表，如 `geek_device_param_all`，不同租户的表名不同
