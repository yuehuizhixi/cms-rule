# 鉴权方案分析 — cms-rule 调用 221 API 的认证方式

> 分析日期：2026-05-12
> 分析范围：business、ems-cloud-service（简称 ems-service）、service-model（hvac-iot）三个微服务

---

## 1. 环境架构

所有微服务部署在 **10.74.170.221**，通过 Kong API 网关统一暴露：
- `/api/business/**` → business 服务 (:7080)
- `/api/cms-cloud-service/**` → ems-service (:8030)
- `/api/hvac_iot/**` → service-model (:8048)

三个服务共用同一个 **Redis**（10.74.170.221:6379），这是鉴权共享的关键。

---

## 2. 现有鉴权机制详解

### 2.1 Business 服务（对外网关层）

#### 2.1.1 两种认证模式

由配置 `com.business.service.poros.authType` 控制：

| 模式 | 描述 | 鉴权方式 |
|------|------|----------|
| `poros` | 对接 Poros 统一认证中心 | 通过 Poros API 获取 accessToken，存 Redis |
| `business` (默认) | 本地登录 | 用户名密码校验，生成 UUID token，存 Redis |

#### 2.1.2 登录流程（business 模式）

1. `POST /poros-authcenter/login` → 校验密码
2. 生成 `LoginResult`，包含 `accessToken`（UUID 字符串）
3. 在 Redis 中记录：`{token} -> "{userId}&{userName}&{tenantId}"`（有效期 36000 分钟）
4. 返回 token 给前端

#### 2.1.3 请求鉴权方式

**SecurityConfiguration** 使用 Spring Security Basic Auth，但实际配置中所有路径都 `permitAll()`：
```java
.antMatchers("/", "/login", "/**").permitAll()
```

真正的鉴权实现在 **`AccessInterceptor`**（HandlerInterceptor）：

- **非登录路径**，调用 `tryLoginFromJwtHeader()`
- 尝试读取请求头 `poros-current-user`，如果是 JWT 格式，解码后自动登录
- 如果用户已登录（SecurityContextHolder 中有 MyUser），但 tenant 变更，也重新登录

#### 2.1.4 `poros-current-user` Header

这是**内部服务间调用**使用的认证头：

- 格式：JWT，使用密钥 `"dev"` 签发（硬编码）
- 解码后得到 `UserBaseInfo`（name, uid, tenantId 等字段）
- 示例：
  ```json
  {"name": "cms-rule", "uid": "system", "tenantId": "geek"}
  ```

#### 2.1.5 Emt-Service 的 AuthHandlerInterceptor 和 License 鉴权

business 和 ems-service 都通过 `AuthConfig` 配置了 `cn.getech.cms.auth` 包提供的 License 管理/多租户隔离：
- `AuthContext` — 许可证/授权上下文
- `AuthHandlerInterceptor` — 请求拦截器，校验许可证
- 由开关 `cn.getech.ems.management.auth.enabled` 控制，默认启用

### 2.2 Ems-Cloud-Service（核心数据层）

#### 2.2.1 用户信息获取

**`EmsUserInfoUtils.getUser()`** 调用 `PorosContextHolder.getCurrentUser()`：

```java
public static UserBaseInfo getUser() {
    UserBaseInfo currentUser = PorosContextHolder.getCurrentUser();
    if (currentUser == null || StringUtils.isAnyBlank(currentUser.getUid())) {
        throw new BusinessErrorException(..., 401);
    }
    // ...
    return currentUser;
}
```

`PorosContextHolder` 是 `cn.getech.poros.framework.common.context` 包中的 **ThreadLocal** 容器，由该包提供的 Filter/Interceptor 自动填充。

#### 2.2.2 鉴权 AOP

`ControllerAspectJ` 切面拦截所有 Controller 方法（除了 `@ExcludeAuth` 标记的），执行：
1. 获取 `PorosContextHolder.getCurrentUser()`→ 无则抛 401
2. 自动注入 `tenantId` 到请求参数对象
3. 检查操作权限（topParent 权限）

#### 2.2.3 `@ExcludeAuth` 免鉴权端点

部分接口（如配置查询、用能数据查询）用 `@ExcludeAuth` 标记，跳过切面鉴权检查。

### 2.3 Service-Model（hvac-iot）

#### 2.3.1 用户信息获取方式（非常关键）

**不使用** `PorosContextHolder`！使用**自定义**的 `UserService.getBaseUserInfo()`：

```java
public static UserBaseInfoDto getBaseUserInfo() {
    HttpServletRequest request = getRequest();
    String authorizationHeaderValue = request.getHeader("Authorization");
    // 也尝试从 Cookie 拿
    if (authorizationHeaderValue != null && authorizationHeaderValue.length() > 7) {
        String token = authorizationHeaderValue.substring(7);
        if (redisTemplate.opsForValue().get(token) != null) {
            String userStr = redisTemplate.opsForValue().get(token);
            String[] uArray = userStr.split("&");
            return new UserBaseInfoDto()
                .setUid(uArray[0])
                .setName(uArray[1])
                .setTenantId(uArray[2]);
        }
    }
    throw new UnauthorizedException();
}
```

**工作原理**：
1. 从请求 `Authorization: Bearer {token}` 中提取 token
2. 用 token 查**共享 Redis**，拿到 `userId&userName&tenantId`
3. 返回用户信息

### 2.4 服务间调用的认证方式

#### 2.4.1 Business → Poros API（Feign）

Business 服务的 `PorosApi`（Feign 客户端）调用 Poros 认证中心时，透传 `Authorization` header。

#### 2.4.2 Business → Ems-Service（内部调用）

未发现显式的 Feign 客户端配置拦截器加头。推测有两种方式：
- 通过 Kong 网关的 `poros-current-user` JWT header 传递
- 或者透传前端的 `Authorization` token

#### 2.4.3 Ems-Service → Business（service-model 等其他服务）

service-model 的 `UserService.getBaseUserInfo()` 直接读 Redis 中的 token 信息。

## 3. cms-rule 需要做什么

### 3.1 目前的情况

cms-rule 需要调用 221 上的三个 API：

| 用途 | URI 前缀 | 认证要求 |
|------|----------|----------|
| Business API（读数据） | `/api/business/**` | `poros-current-user` JWT header 或 `Authorization` token |
| CMS-Cloud API | `/api/cms-cloud-service/**` | 需要 `PorosContextHolder` 有用户信息 |
| HVAC-IoT API（下发指令） | `/api/hvac_iot/**` | 需要 Redis 中有对应的 token 记录 |

**当前实现**：
- `EmsParamService` 通过 `restTemplate` 调用本地代理端点（`/api/rule-engine/proxy/...`）
- 携带 `cms-rule.proxy.auth-token` 作为 `Bearer` token
- `CmdClient`（Feign）直接调用 `/api/hvac_iot/cmd`，**未配置认证拦截器**

### 3.2 关键发现：221 的实际认证要求

经过代码分析，**221 上的三个微服务认证的共享机制是 Redis**：

1. **business 服务**：`AccessInterceptor` 也支持 `poros-current-user` header（JWT）
2. **ems-service**：通过 `poros-framework` 的 Filter 从 `poros-current-user` header 读取并设置 `PorosContextHolder`，或 `Authorization` token 查 Redis
3. **service-model**：直接读 `Authorization` header 中的 token，查 **共享 Redis**

**所以服务间调用不需要走 Poros 认证中心的完整 OAuth 流程**。

### 3.3 推荐方案：使用 JWT 作为 `poros-current-user` header

在 cms-rule 的 `EmsParamService` 和 `CmdClient` 调用时，**生成一个 JWT 放在 `poros-current-user` header 中**，这样所有三个服务都能识别：

- business 的 `AccessInterceptor.tryLoginFromJwtHeader()` 会解析并登录
- ems-service 的 `PorosContextHolder` Filter 会解析并设置上下文
- service-model 也可以通过相同方式处理

#### 3.3.1 方案一：直接透传 proxy.auth-token（已有实现）

当前 `EmsParamService.postForMap()` 已经实现了通过 `Authorization: Bearer {proxyAuthToken}` 调用本地代理端点。但这个 token 是一个自定义 JWT，service-model 的 `UserService.getBaseUserInfo()` 无法用它查 Redis（Redis 中没有记录）。

**所以当前方案仅对 business 的免鉴权接口有效**，对于 service-model 的下发指令接口不可行。

#### 3.3.2 方案二（推荐）：在 Feign/RestTemplate Interceptor 中添加 `poros-current-user` header

生成一个合法的 JWT，内容包含 `name`、`tenantId`、`uid` 等信息，使用 business 服务约定的 JWT 密钥（`"dev"`）签名。

**JWT 内容示例**：
```json
{
  "sub": "cms-rule-service",
  "userName": "cms-rule",
  "userId": "system",
  "tenantId": "geek",
  "iat": ...,
  "exp": ...
}
```

将这个 JWT 放在 `poros-current-user` header 中发送。

#### 3.3.3 方案三（备用）：将 token 写入 Redis

在 cms-rule 启动时，将 `proxy.auth-token` 按 business 服务的格式写入 221 的 Redis：
```
SET {token} "{userId}&{userName}&{tenantId}"
EXPIRE {token} 36000
```

这样 service-model 的 `UserService.getBaseUserInfo()` 就能通过共享 Redis 查到用户信息。

## 4. 具体实现方案

### 4.1 CmdClient Feign Interceptor 实现

在 Feign 客户端添加 RequestInterceptor，自动添加认证头：

```java
// FeignAuthInterceptor.java
@Component
public class FeignAuthInterceptor implements RequestInterceptor {
    
    @Value("${cms-rule.proxy.auth-token}")
    private String authToken;
    
    @Override
    public void apply(RequestTemplate template) {
        // 使用 poros-current-user header（JWT 格式）
        // 这样 business/ems-service/service-model 都能识别
        template.header(HttpHeaders.AUTHORIZATION, "Bearer " + authToken);
    }
}
```

### 4.2 RestTemplate Interceptor 实现

```java
// RestTemplateAuthInterceptor.java 或 EmsParamService 中直接设置
headers.set(HttpHeaders.AUTHORIZATION, "Bearer " + proxyAuthToken);
```

### 4.3 给所有 Feign 客户端应用 Interceptor

**方式 A**：全局配置类

```java
@Configuration
public class FeignConfig {
    
    @Bean
    public RequestInterceptor authRequestInterceptor(
            @Value("${cms-rule.proxy.auth-token}") String authToken) {
        return requestTemplate -> {
            // 如果关键路径不需要认证头，可以加条件判断
            requestTemplate.header("poros-current-user", createPorosJwt());
        };
    }
    
    private String createPorosJwt() {
        // 用 JWT 工具生成，使用密钥 "dev"
        // 内容：name=cms-rule, uid=system, tenantId=geek
        ...
    }
}
```

**方式 B**：每个 FeignClient 单独配置

```java
@FeignClient(
    name = "cmd-service", 
    url = "${cms-rule.cmd.remote-url}",
    configuration = CmdClientConfig.class
)
public interface CmdClient { ... }

public class CmdClientConfig {
    @Bean
    public RequestInterceptor cmdAuthInterceptor() {
        return template -> {
            // 使用 proxy-auth-token 作为 Bearer token
            // 或者使用 poros-current-user JWT
        };
    }
}
```

### 4.4 JWT 生成工具

```java
@Component
public class PorosJwtGenerator {
    
    private static final String JWT_SECRET = "dev";
    
    public String createServiceToken() {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userName", "cms-rule");
        claims.put("userId", "system");
        claims.put("tenantId", "geek");
        
        return Jwts.builder()
            .setClaims(claims)
            .setSubject("cms-rule-service")
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + 3600_000 * 24 * 365)) // 1年
            .signWith(SignatureAlgorithm.HS256, TextCodec.BASE64.encode(JWT_SECRET))
            .compact();
    }
}
```

## 5. 总结

| 服务 | 鉴权方式 | cms-rule 如何认证 |
|------|----------|------------------|
| **business** | `AccessInterceptor` 读 `poros-current-user`（JWT） | ✅ 已通过 proxy.auth-token + Bearer 实现部分 |
| **ems-cloud-service** | `PorosContextHolder` 读 `poros-current-user`（JWT）或 `Authorization` | ✅ 需要在 proxy 透传 `Authorization` header |
| **service-model** (hvac-iot) | `UserService.getBaseUserInfo()` 读 `Authorization` token → 查 Redis | ❌ 需要持有有效的 Redis token，或转为 JWT 方案 |

**推荐**：最简方案是在 Feign 层统一添加 `Authorization: Bearer {proxy-auth-token}` header。如果 service-model 的 Redis 查不到 token，则需要同步写入 Redis，或者改为使用 JWT 的 `poros-current-user` 方案（需要确认 `poros-framework` 在 221 上是否已部署对应 Filter）。

**当前风险**：`CmdClient` Feign 调用 `/api/hvac_iot/cmd` 时**未携带任何认证头**，而 service-model 的 `UserService.getBaseUserInfo()` 会从 `Authorization` 头解析 token 查 Redis。如果 token 不存在，会抛出 `UnauthorizedException`（HTTP 401）。

### 5.1 推荐优先级

1. **立即**：给 `CmdClient` 添加 `Authorization: Bearer {token}`，将 proxy.auth-token 作为 Bearer token 传递（已在 `application.yml` 中配置，但 CmdClient 未使用）
2. **验证**：确认 service-model 的 Redis 中已有该 token 记录（否则需要写入），或确认 `poros-current-user` header 的处理链路是否完整
3. **长期**：统一使用 JWT（`poros-current-user`）作为服务间调用凭证，避免依赖 Redis 共享
